🎯 Problem Statement

The Business Context

Opendoor (real estate platform) sends 1M letters/month to homeowners. Each letter costs:

  • $0.10 - Printing
  • $0.12 - Postage (First Class)
  • $0.06 - Operational overhead
  • = $0.28 per letter

Data Matching Challenge

┌─────────────────────────────────┐ │ USER ADDRESS (Opendoor DB) │ │ "123 Main St, │ │ Springfield, IL 62701" │ └──────────────┬──────────────────┘ │ ❌ MISMATCH │ ┌──────────────▼──────────────────┐ │ POSTAL DB ADDRESS │ │ "123 Main Street, │ │ Springfield, Illinois 62701" │ └─────────────────────────────────┘ Issues Found: ✗ "St" vs "Street" (abbreviation) ✗ "IL" vs "Illinois" (state format) ✗ Missing ZIP+4 (62701-1234)

Root Causes of Mismatches (19%)

  • Abbreviations: "St" vs "Street", "Ave" vs "Avenue" (28% of errors)
  • Typos: "Main" vs "Mian", "Sprinfield" (19% of errors)
  • State Format: "IL" vs "Illinois" (15% of errors)
  • Missing Components: No ZIP+4, apt/suite numbers (18% of errors)
  • Case Sensitivity: "MAIN" vs "Main" (8% of errors)
  • City Name Variations: "St Louis" vs "Saint Louis" (12% of errors)

🏗️ Architecture & Agentification Strategy

Why Agentification?

Traditional rule-based systems are brittle and unmaintainable. Agentic AI allows the system to:

  • Reason about ambiguities - Not just match, but understand context
  • Use multiple data sources - Call tools iteratively
  • Handle edge cases - Complex situations require reasoning
  • Improve over time - Learn from feedback
  • Explain decisions - Transparency for manual review

🧠 Algorithm Deep Dive

1. Fuzzy String Matching (Jaro-Winkler)

Compares strings accounting for typos and transpositions.

# Example: Comparing street names "MAIN" vs "MIAN" → Jaro-Winkler score: 0.92 (likely same) "STREET" vs "ST" → Jaro-Winkler score: 0.76 (fuzzy match) "123 Main St" vs "123 Main Street" → After normalization: ~0.95

2. Phonetic Matching (Soundex)

Handles sound-alike addresses.

# Soundex example: "SMITH" → S500 "SMYTH" → S500 (sound identical!) Match: ✓ "SPRINGFIELD" → S165 "SPRINFIELD" → S165 (typo doesn't affect phonetic) Match: ✓

3. Abbreviation Standardization

Abbreviation Full Form Frequency
St Street 28%
Ave Avenue 15%
Blvd Boulevard 8%
IL Illinois 85%
CA California 82%

4. Component Weighting

Final Score = Σ(component_score × weight) Weights: - Street Number: 20% (must match exactly) - Street Name: 35% (most variable) - Street Type: 10% (often abbreviated) - City: 15% (sometimes varies) - State: 10% (must match) - ZIP Code: 10% (most reliable) Example: "123 Main St, Springfield, IL 62701" vs "123 Main Street, Springfield, IL 62701-1234" Scores: - Street Number (123 = 123): 1.0 × 0.20 = 0.20 - Street Name (Main = Main): 1.0 × 0.35 = 0.35 - Street Type (St = Street): 0.8 × 0.10 = 0.08 - City (Springfield = Springfield): 1.0 × 0.15 = 0.15 - State (IL = IL): 1.0 × 0.10 = 0.10 - ZIP (62701 = 62701): 0.7 × 0.10 = 0.07 ──────────────────────────────────────── TOTAL SCORE: 0.95 ✓ MATCH

5. Confidence Thresholds

Score Confidence Level Action Latency Cost
≥ 0.95 HIGH Deliver immediately 50ms $0.002
0.80-0.95 MEDIUM Validate with USPS/tax records 200ms $0.02
< 0.80 LOW Use agent reasoning or manual review 2000ms $0.05

🎓 L5 Interview Questions

Question 1: How would you design this system from scratch?

Q: Walk me through how you'd architect address matching for 1M monthly letters.
A: I'd start with the business constraint: minimize misdeliveries while keeping latency reasonable. 1. REQUIREMENTS ANALYSIS: - 1M addresses/month = ~40k/hour - Cost per misdelivery = $0.28 - Accept latency of 100-200ms per address - Need to handle typos, abbreviations, format variations 2. APPROACH: Layer 1 (99%): Vector embeddings - Fast, cheap ($0.002), catches straightforward matches - ~50ms latency Layer 2 (0.9%): Validation APIs - USPS, tax records, geocoding - Increases confidence for medium-case matches - ~200ms latency Layer 3 (0.1%): Agent reasoning - Claude with tool use for edge cases - Iteratively reason about contradictions - ~2s latency, but only for 1% of addresses 3. TECHNOLOGY: - Python for agent logic - PySpark for batch processing 1M addresses - Pinecone for vector search (fast) - Neo4j for knowledge graph (relationships) - Redis for caching (80% repeated queries) 4. EXPECTED RESULTS: - 91% accuracy (up from 65%) - 85ms average latency - $5k/month cost → $100k+/month savings - ROI: 17x

Question 2: Why use agentification here?

Q: This could be solved with simple fuzzy matching. Why add Claude Agent complexity?
A: Good question. Initially I thought the same. But here's why agents are critical: 1. EDGE CASES (1% of addresses): - Simple fuzzy matching fails on: * "123 Main St #4" vs "123 Main Street Apt 4" * "St. Louis" vs "Saint Louis" * Coordinate mismatch (wrong ZIP) but correct property - Agent can reason: "Coordinates match property record, must be correct despite ZIP discrepancy" 2. MULTI-STEP REASONING: - Agent calls USPS API → No ZIP+4 found - Agent calls tax records → Property found at this address - Agent calls geocoding → Coordinates match perfectly - Agent decides: "Match with confidence 0.98, note: missing ZIP+4" - Fuzzy logic can't make this decision 3. HANDLING CONTRADICTIONS: - What if USPS says valid but geocoding says wrong coordinates? - Agent can reason about why (e.g., USPS might have old data) - Make intelligent decision based on evidence weight 4. COST/BENEFIT TRADE-OFF: - Agent slows down 1% of addresses (2s vs 50ms) - But catches 26% more addresses correctly - Saves $72.8k/month in misdeliveries - Agent cost: only $2-3k/month for 1% of traffic DECISION: Use agent selectively (fallback only). This gets 99% speed of fast path with 99.5% accuracy.

Q: What could go wrong? How would you handle it?
A: Several failure modes to consider: 1. USPS API OUTAGE: Problem: 20% of addresses use USPS validation layer Impact: Would fall back to embedding-only (lower accuracy) Solution: Cache USPS results (30-day TTL) Fallback to tax records + geocoding Cost: Accuracy drops from 91% to 87% temporarily 2. VECTOR DB (PINECONE) DOWN: Problem: 99% of addresses use fast path Impact: Cannot match any addresses Solution: Fallback to simple Levenshtein distance (less accurate) Would process manually next day Cost: ~$35k in one day of mismatches 3. KNOWLEDGE GRAPH CORRUPTION: Problem: Bad data in Neo4j → incorrect matches Impact: ~5% match rate wrong due to corrupted relationships Solution: Weekly integrity checks Rollback to previous week backup Cost: ~$5k in mismatches until fixed 4. AGENT TIMEOUT (Claude unavailable): Problem: Can't process 1% of edge cases Impact: ~10k addresses flagged for manual review Solution: Queue for next day processing Send follow-up letters once matched Cost: ~$2.8k in delayed mailings (acceptable) 5. COST SPIKE: Problem: Claude agent called for too many addresses (budget overage) Impact: $20k/month cost instead of $5k Solution: Implement cost guardrails Alert if agent called >10k times/day Automatically cap at budget Cost: Accuracy drops from 91% to 88% 6. DATA QUALITY ISSUE: Problem: User database has many incomplete addresses Impact: Can't match what isn't there Solution: Validation layer catches incomplete addresses Flag for user to provide complete address Cost: ~1% addresses need correction MONITORING STACK: - Pinecone: SLA monitoring + fallback - USPS: Rate limiting + retry logic - Agent: Budget cap + latency monitoring - Overall: Daily accuracy report + anomaly detection If accuracy drops >2%: Alert ops team immediately.

Question 6: Trade-off deep dive

Q: You said you accept 0.1% manual review. Why not optimize to 0.01%?
A: Great question about diminishing returns. CURRENT STATE (0.1% manual): - 1M addresses → 1,000 need manual review - Manual cost: $5/address (1 min at $300/hr) = $5k/month - Total system cost: $50k + $5k = $55k/month - Savings: $100k/month - ROI: 82% OPTIMIZING TO 0.01% (90% more perfect): To get from 0.1% to 0.01%: - Call Claude agent for all unconfident matches (instead of 1%) - Now agent called for 5-10% of addresses (50-100k/day) - Cost: $0.05 × 500k addresses = $25k additional - Total system cost: $50k + $25k = $75k/month - New savings: $100k + $0.28 × 900 = $100.25k/month - ROI: 34% CONCLUSION: - Going from 0.1% to 0.01% costs 50% more - But only saves $0.25k/month - ROI drops from 82% to 34% - It's not worth it THE RULE: "Optimize until ROI starts dropping, not until perfection." We pick 0.1% because: ✓ ROI is still 82% (excellent) ✓ Manual review is manageable (1k/month) ✓ Edge cases are infrequent ✓ Cost-benefit clearly favors this level This is classic optimization trade-off analysis.
