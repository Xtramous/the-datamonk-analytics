1
🚀 Ingestion Layer (Event Streaming)

Centralized event collection, validation, and retention

Message Broker
Apache Kafka
🎯 Distributed message queue handling 100K events/min, partitioned by metric_id (isolate hot/cold metrics)
Partitioning: metric_id key
Retention: 7 days (enable replay)
Replication: 3x (HA)
Cost: $3K/month
Schema Registry
Confluent Schema Registry
🎯 Enforces schema contract between producers/consumers, versioning for metric schema evolution
Format: Avro/Protobuf
Validation: Producer/consumer compatibility checks
Version Control: Enable replay with old schemas

⚡ PARALLEL PROCESSING PATHS (From this point onward)

🟢 REAL-TIME PATH (Stream Processing)

Low-latency aggregation for dashboards (1-2 min to display)

2A
⚙️ Stream Processing Engine

Stream Processor
Apache Flink
🎯 Real-time aggregation: 15-30 second tumbling windows, SUM/COUNT/PERCENTILES (p50, p95, p99)
Windows: 15-30 seconds
State: RocksDB (fault-tolerant KV store)
Output: Flush to Redis every 1 minute
Cost: $8K/month (8-12 task managers)
State Backend
RocksDB
🎯 Embedded KV store for Flink state (aggregation results), survives restarts via incremental checkpoints
Embedded in Flink: No separate deployment
Checkpointing: Incremental (faster recovery)
Backend: Local SSD or HDFS
↓

2A-S
💾 Hot Storage (Real-time)

Time-Series Cache
Redis TimeSeries
🎯 In-memory time-series data (7 days), <5ms reads for real-time dashboards, auto-expire after 7 days
Capacity: 350-500GB on-disk
Latency: <5ms
Key Format: metric:{id}:{dim1=val1}:{dim2=val2}:{ts}
TTL: 7 days auto-expiry
Cost: $5K/month

🔵 BATCH PATH (Correctness)

Full-day re-aggregation for historical accuracy (nightly)

2B
⚙️ Batch Processing Engine

Batch Processor
Apache Spark
🎯 Nightly batch: full day's Kafka events, 24-hour late-arrival dedup window, exactly-once semantics for revenue metrics
Schedule: Nightly (2 AM UTC)
Late Window: 24 hours
Writes: Idempotent (UPSERT)
Output: Snowflake fact tables
Language: PySpark/Scala
↓

2B-S
💾 Cold Storage (Batch)

Data Warehouse
Snowflake
🎯 2-year historical data, star schema (metrics_fact + dimensions), <500ms analytical queries, vectorized execution
Schema: Star (fact + dim tables)
Partitioning: By date, nested by metric_id
Format: Native Snowflake
Cost: $6K/month (2 XL clusters, auto-suspend off-peak)
Columnar Format
Parquet + Snappy
🎯 Compressed columnar format for S3 archives (10:1 ratio), enables Delta Lake ACID transactions
Compression: 10:1 ratio
Storage: S3
Format: Delta Lake for ACID
Archive Storage
AWS S3 + Glacier
🎯 Long-term storage: S3 for 90 days (warm), Glacier for 2+ years (cold archive, rarely accessed)
Tiering: S3 → Glacier after 90 days
Cost: $0.01/GB/month (Glacier)
Retention: 2+ years

dbt (data build tool)

Schedule: Nightly (after Spark batch)
Tests: Null counts, freshness, relationships
Output: Snowflake native tables
Language: Jinja2 + SQL
Version Control
Git (GitHub/GitLab)
🎯 Version control for dbt models, SQL transformations, schema documentation, reproducible analytics
Repo: dbt models + tests
CI/CD: dbt Cloud or GitHub Actions
Lineage: Automatic tracking

4
⚡ Multi-Layer Caching

Query result caching for 70% hit rate on repeated queries

Query Cache
Memcached
🎯 Query result cache: caches full query responses (5min TTL), 70% hit rate (users repeat same metric slices)
TTL: 5 minutes
Hit Rate: 70%
Key Format: hash(metric_id+dims+range)
Invalidation: Event-driven on Flink flush
Hot Data Cache
Redis (as cache)
🎯 Dual duty: both hot storage (7d retention) AND distributed cache for architectural efficiency
TTL: 7 days (data)
Operations: GET, SET, EXPIRE
Invalidation: Check invalidation log on query

🔌 Query APIs

Three APIs optimized for different client needs

REST API
FastAPI / Flask
🎯 Fast Query API (<100ms): GET /metrics/{id}?dimensions=x,y&time_range=7d → hits Redis directly
Framework: FastAPI (async Python)
Response: JSON
SLA: <100ms p99
Cache: Memcached layer
SQL API
Analytical Query Service
🎯 Analytical API (<500ms): POST /query with complex filters/joins → routes to Snowflake cold path
Query Builder: Parse JSON → Snowflake SQL
Optimizer: Detect filter selectivity
SLA: <500ms p99
GraphQL API
GraphQL Server
🎯 Batching API: combine N independent metric requests into 1 Snowflake query (N calls → 1 query, reduce round-trips)
Framework: Apollo Server
Feature: Field aliasing (revenue_usd → gross_revenue - taxes)
Batching: DataLoader pattern

🎯 Key Insights from Complete Stack

  • Dual Architecture: Real-time path (Flink → Redis) for dashboards, batch path (Spark → Snowflake) for correctness. Never compromise latency or accuracy.
  • Parallel Processing: Flink and Spark run independently on same Kafka source. Flink feeds dashboards in 1-2 min, Spark provides historical record nightly.
  • Multi-Layer Caching: Memcached (query results) + Redis (hot data) + dbt (materialized views) = 70% cache hit rate, 10x query speedup for golden metrics.
  • Separation of Concerns: Each tool does ONE thing well. Kafka: streaming, Flink: low-latency agg, Spark: batch correctness, Snowflake: analytical queries, Redis: fast reads.
  • Cost Optimization: $22K/month for 1000 metrics = $22/metric. At 5000 metrics, unit cost drops to $7/metric (economies of scale).
  • Operational Excellence: Prometheus → Grafana → PagerDuty pipeline ensures sub-minute MTTR (Mean Time To Resolution).
