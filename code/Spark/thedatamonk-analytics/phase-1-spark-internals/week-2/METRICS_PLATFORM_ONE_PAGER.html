📊 Metrics Platform - System Design One-Pager

1000+ metrics, 100 concurrent users, <100ms real-time, <500ms analytical, 2-year retention

🏗️ ARCHITECTURE

📱 Clients → 🔌 Query APIs → 💾 Storage → ⚙️ Compute → 🚀 Ingestion

INGESTION LAYER

Kafka: 100K events/min, partition by metric_id
Schema Registry: Version control + validation
Retention: 7 days (enable replay)
Why Kafka vs Kinesis? Cost ($3K vs $10K/mo) + portability

COMPUTE LAYER

Real-time (Flink): 15-30s windows, RocksDB state, 1min flush to Redis
Batch (Spark): Nightly re-aggregation, exactly-once, late arrivals
Why dual engines? Flink optimized for latency, Spark for batch correctness

STORAGE LAYER

HOT (7d): Redis TimeSeries (350GB)
Latency: <5ms | Key: metric_id:dim:timestamp
COLD (2yr): Snowflake + Delta Lake (S3)
Latency: <500ms | Star schema + Parquet (10:1 compression)
MATERIALIZED: dbt (50 golden metrics, hourly + daily)
99% of queries hit pre-agg data: 500ms → <50ms

QUERY LAYER - 3 APIs

Fast API<100msRedisDashboards
Analytical<500msSnowflakeAd-hoc queries
GraphQL<500msBatchedBI tools

CACHING LAYER

Memcached: Query results (5min TTL, 70% hit rate)
Redis: Double duty as hot storage + cache
Invalidation: Event-driven, no thundering herd

🎯 TRADEOFFS EXPLAINED

Dual Storage (Hot + Cold):
✓ Achieve <100ms AND 2-year retention
✗ Operational complexity, data sync overhead
Kafka vs Kinesis:
✓ Save $7K/month, portable
✗ Self-managed ops overhead
Dual Engines (Flink + Spark):
✓ Optimize latency + correctness
✗ Dual codebases, more infra
Eventual Consistency (5min lag):
✓ Buy latency by accepting lag
✗ Revenue metrics: nightly reconciliation >1% alerts
Materialized Views:
✓ 99% queries pre-aggregated
✗ Forecast which 50 metrics matter (non-obvious at 1000)

❌ WHY NOT SINGLE SYSTEM?

BigQuery alone? Can't do <100ms cold storage
Redshift alone? Same issue + lower throughput
Single Spark? Micro-batches = higher latency
Single Flink? Memory inefficient for 100GB+ batch

📋 RELIABILITY

Exactly-Once

Spark: Idempotent writes (upsert), 24hr dedup window
Flink: Kafka checkpointing + transactional Redis writes

Replay & Backfill

Schema changes? Registry version captures it
Job failed? Re-run Spark with --date-range
Logic bug? Kafka 7d retention enables replay

Data Quality

Flink: Real-time validation (metric_value > 0?)
Snowflake: dbt tests (nulls, freshness, relationships)
Alerting: Late data >1hr lag → PagerDuty

🎤 INTERVIEW NARRATIVE

"Core challenge: real-time (<100ms) and historical (2yr) conflict. Real-time needs sub-100ms—impossible with Snowflake alone. Historical needs 2yr + flexibility—Redis can't store cheaply.

Solution: Dual storage. Redis (7d hot, <5ms), Snowflake (2yr cold, <500ms).

Ingestion: Kafka (replay capability) + Schema Registry (contract enforcement).

Compute: Flink (15-30s windows → Redis 1min) for real-time. Spark batch nightly for correctness + late arrivals.

Query: 3 APIs. Fast (Redis), Analytical (Snowflake), GraphQL (batching).

Biggest tradeoff: Eventual consistency (5min lag). Redis/Snowflake diverge, nightly reconciliation catches >1% discrepancies. Deliberate: buy latency over sync complexity.

At 100 users, 1000 metrics: ~$22K/mo. At 5000 metrics: shift 80% to Snowflake, keep top 200 in Redis."

❓ FOLLOW-UPS PREPARED

Why not BigQuery? Can't achieve <100ms cold storage; tiered approach necessary.
Schema evolution? Registry versioning + Flink schema-on-read + Spark multi-version backfill.
Kafka down? App buffers 1hr locally; Kafka recovery <10min; dashboards fallback to stale 7d Redis cache.
Test dual paths? dbt tests (Snowflake) + Flink unit tests + integration tests comparing tail records.
Hot metric isolation? Kafka partition by metric_id prevents one hot metric blocking others.
Handling skew? Materialized views + pre-aggregation removes per-query load imbalance.
Why not Kinesis/BigQuery? Vendor lock-in + can't achieve <100ms + costs more. Kafka/Snowflake open + cheaper + flexible.

✨ REMEMBER THIS

Golden Principle: Separate concerns by latency requirements. Real-time path optimized for speed, batch path optimized for correctness. Never force one path to do both.
Key Insight: Eventual consistency is not a bug, it's a feature. Nightly reconciliation + alerts catch issues without synchronous complexity.
Cost Multiplier: Materialized views (pre-aggregation) = 10x cheaper than computing per-query for 70% hit rate.
