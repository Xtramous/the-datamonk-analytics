1000+ metrics, 100 concurrent users, <100ms real-time, <500ms analytical, 2-year retention

🏗️ SYSTEM ARCHITECTURE

Architecture Flow

1. INGESTION LAYER (Kafka)

Throughput: 100K events/minute = 1,667 events/second
Partitioning: metric_id (isolates hot metrics from cold, e.g., "revenue" vs "page_views_on_mobile_in_japan")
Schema Registry: Enforces compatibility between producers/consumers. Prevents parsing failures when metric schema evolves (critical at 1000+ metrics)
Retention: 7 days enables replay if Flink/Spark job fails
Cost Decision: Kafka ($3K/mo, 6 brokers) vs Kinesis ($10K/mo). Tradeoff: ops overhead vs $7K/month savings

2. COMPUTE LAYER (Dual Engines)

Real-time Path: Apache Flink
Windows: 15-30 second sliding windows (granular aggregation)
State Backend: RocksDB (embedded KV store, fault-tolerant)
Aggregations: SUM, COUNT, PERCENTILES(p50, p95, p99)
Output: Flushed to Redis every 1 minute (batched writes for efficiency)
Latency: Event arrival → Flink aggregation → Redis ≈ 1-2 minutes to dashboard
Batch Path: Apache Spark
Schedule: Nightly (e.g., 2 AM UTC) processes full day's Kafka events
Late Arrivals: 24-hour deduplication window (catches events arriving >1hr late)
Exactly-Once: Idempotent writes (SUM, COUNT use database UPSERT) prevent double-counting
Output: Writes to Snowflake fact tables (immutable, append-optimized)
Why Dual? Flink optimized for low-latency (streams), Spark optimized for batch correctness. Single engine can't excel at both.

3. STORAGE LAYER (Tiered)

HOT PATH (0-7 days)

Technology: Redis TimeSeries Cluster (RTS module)
Capacity: 350-500GB on-disk (7 days × 1000 metrics × 100 dimensions × ~3KB per series)
Latency: <5ms reads (pure RAM with RDB persistence)
Key Structure:
metric:{metric_id}:{dim1=val1}:{dim2=val2}:{timestamp}
Example: metric:revenue:region=US:product=pro:1718821200
TTL: Automatic expiration after 7 days (Redis EXPIRE)
Query: O(1) time-range + dimension filters via ZRANGE with predicates

COLD PATH (7 days - 2 years)

Technology: Snowflake + Delta Lake (S3 Parquet)
Schema: Star schema
• Fact: metrics_fact (metric_id, timestamp, dim1_id, dim2_id, value)
• Dimensions: dim_date, dim_product, dim_geography, etc.
Partitioning: By date (2024-06-19, 2024-06-20, ...), nested by metric_id
Compression: Parquet + Snappy = 10:1 ratio (100GB raw → 10GB compressed)
Latency: <500ms (vectorized query execution, columnar scan efficiency)
Retention: Full 2 years; move to Glacier after 90 days (cheaper storage, slower access OK for archive)

MATERIALIZED VIEWS (dbt)

Golden Metrics (Top 50): revenue, orders, margin, CSAT, attribution, churn, etc.
Refresh: Nightly (after Spark batch completes)
Granularity: Pre-aggregated at 1-hour and daily level (faster than raw facts)
Storage: Snowflake native + hot subset synced to Redis
Impact: 99% of queries hit pre-agg data: 500ms query → <50ms

4. QUERY LAYER (3 APIs)

Fast Query API (<100ms SLA)

Use Case: Real-time dashboards (sales dashboard, ops monitoring)
Endpoint: GET /metrics/{metric_id}?dimensions=geography,product&time_range=7d
Storage: Redis TimeSeries (direct hit on hot cache)
Query Logic: Time-range + dimension filter in O(1) with Redis ZRANGE + predicates
Example Response:
{
  "metric": "revenue",
  "dimensions": {"region": "US", "product": "Pro"},
  "data": [
    {"timestamp": 1718821200, "value": 50000},
    {"timestamp": 1718821260, "value": 51200}
  ]
}
Response Caching: Result cached 5min in Memcached. Invalidated on new Flink flush.

Analytical Query API (<500ms SLA)

Use Case: Data explorers, BI tools, ad-hoc analysis
Request Example:
POST /query
{
  "metrics": ["revenue", "orders", "margin"],
  "dimensions": ["date", "geography", "channel"],
  "filters": {"geography": ["US", "EU"], "channel": "direct_mail"},
  "granularity": "daily",
  "time_range": "90d"
}
Query Optimizer: Detects filter selectivity. If filtering results in <1M rows, runs on hot storage (Redis). Else, routes to Snowflake cold path.
Result Caching: Large result sets (100M rows → 10MB compressed) cached in Memcached, 5min TTL

GraphQL Layer (BI Integrations)

Use Case: Tableau, Looker, Power BI integrations
Batching: Combines N independent metric requests into 1 Snowflake query (N calls → 1 query)
Field Resolution: Handles aliases (revenue_usd → gross_revenue - taxes - refunds)
Benefit: Reduces round-trip latency significantly for dashboard fetches
Why 3 APIs? Different tools, different needs. Dashboards tolerate <100ms. BI tools pre-fetch reports, can wait 500ms. GraphQL for integrations needing efficient batching.

5. CACHING LAYER

Multi-Layer Cache Architecture

LayerTechnologyTTLHit Rate
L1: Query Results Memcached 5 min 70%
L2: Hot Data Redis TimeSeries 7d (TTL) 100%
L3: Cold Data Snowflake Cache 1d 40%

Cache Invalidation Strategy

Event-Driven: When Flink outputs new data (1min), publishes cache invalidation event
Invalidation Event Format: {metric_id, timestamp, affected_dimensions}
Query API Logic: Checks invalidation log before serving from Memcached. If hit invalidation timestamp after cache write, miss and recompute.
No Thundering Herd: Stagger cache refresh across 50 replicas (each replica refreshes different key range)
Tradeoff: Cache coherence is hard. Alternative: no explicit invalidation, just TTL expiry (simpler but 5min stale reads). For revenue metrics, explicit invalidation is necessary.

Cache Efficiency Metrics

70% hit rate: Users ask same questions repeatedly (revenue by region, top products, etc.). Pre-agg golden metrics amplify hits.
70% cache hit = 3x effective throughput (cache serves instantly, DB saves compute)

Tradeoffs, Reliability, Scaling, Costs, and Interview Narrative

⚖️ KEY TRADEOFFS

1. DUAL STORAGE (Hot + Cold)
✓ Achieve <100ms latency (Redis) + 2-year retention (Snowflake)
✗ Operational complexity (manage 2 systems), data sync overhead (eventual consistency)
💡 Why: Impossible to achieve both with single system. Snowflake alone can't do <100ms. Redis can't store 2yr cheaply.
2. KAFKA vs KINESIS
✓ Kafka: $3K/mo, portable (multi-cloud), self-managed
✗ Operational overhead (6-9 node HA cluster, monitoring)
✓ Kinesis: Fully managed (AWS)
✗ $10K/mo, vendor lock-in
💡 Why: At 100K events/min, $7K/month savings justify ops overhead. K8s automation makes Kafka manageable.
3. FLINK + SPARK (Dual Compute)
✓ Flink for low-latency streams (15-30s windows), Spark for batch correctness
✗ Dual codebases, more infrastructure to manage, learning curve
✓ Single Spark: simpler, one codebase
✗ Micro-batches → higher latency (>1min), not suitable for dashboards
💡 Why: Can't sacrifice latency. Separate engines optimize for their strengths.
4. EVENTUAL CONSISTENCY (5min lag)
✓ Accept 5-min lag between Redis and Snowflake (no sync complexity)
✗ Temporary data divergence (revenue in Redis may differ from Snowflake)
✓ Alternative: synchronous writes (strong consistency)
✗ Distributed transaction complexity, latency adds up
💡 Why: For most metrics (pageviews, clicks), 5min lag acceptable. For revenue: nightly reconciliation flags >1% discrepancies + alerts.
5. MATERIALIZED VIEWS (Pre-aggregation)
✓ Pre-compute 50 golden metrics (revenue, orders, CSAT, attribution) → 10x query speedup
✗ Requires forecasting which metrics matter (non-obvious with 1000+ metrics)
✗ Maintenance burden (dbt jobs, schema evolution)
💡 Why: 70% of queries hit 50 metrics. Pre-agg reduces 500ms → <50ms for those. Covers 80/20 rule.

🔒 RELIABILITY

Exactly-Once Semantics (Revenue Metrics)

Critical: Revenue, ad spend, refunds must never be double-counted
Spark Path: Idempotent aggregations. SUM/COUNT use database UPSERT (overwrite, not append). 24-hour deduplication window catches late arrivals.
Flink Path: Kafka offset checkpointing (stores processed offset to RocksDB). If Flink restarts, resumes from saved offset, skips already-processed events.

Backfilling & Replay

Schema Changes: Schema Registry versions captured. If event schema evolves (e.g., added new dimension), Flink/Spark can handle multiple versions simultaneously.
Job Failure: Kafka 7-day retention enables re-processing. Spark replay: spark-submit ... --date-range 2024-06-18 2024-06-19
Logic Bug: If aggregation logic bug discovered, replay Spark job with corrected logic. Redis data already expired (7d TTL), so stale data doesn't linger.

Data Quality

Flink Validation: Real-time checks (metric_value > 0? revenue > 0?). Bad events dropped + alert to ops.
Snowflake Tests (dbt): Null counts, freshness (latest record within 1hr?), referential integrity (dim_id exists in dimension table?).
Alerting: Prometheus metrics on late-arriving data (>1hr lag → PagerDuty). Reconciliation job flags >1% Redis/Snowflake discrepancy.

Disaster Recovery

Kafka Down (<10min expected): Apps buffer events locally for 1 hour. Dashboards show 7-day cached data (stale but available). Resume normal flow on recovery.
Redis Failure: Flink can re-flush last 1 minute of data. Alerts notify team. Cold path (Snowflake) always available.
Snowflake Failure: Rare (SLA 99.9%). Cold path unavailable, but hot path (7d Redis) continues. Historical queries degrade to Redis-only for those 7 days.

📈 SCALING PLAN

Current State (1000 metrics)

Redis Memory: ~350GB (comfortable, room to grow)
Flink Tasks: 8-12 task managers (auto-scale on kafka_consumer_lag)
Snowflake Compute: 2 XL clusters (auto-suspend during off-peak)
Throughput: 100K events/min easily handled

At 5000 Metrics (12 months)

Redis: Shift 80% of hot data → Snowflake hot cluster (faster), keep only top 200 golden metrics in Redis
Reasoning: 4000 metrics don't need <5ms access. 1hour Snowflake cluster sufficient. 30% cost savings.
Flink: 20-30 task managers. Partition rebalancing handled automatically by Kafka consumer groups.
Spark: Add intermediate hourly aggregation layer (pre-agg top 500 metrics hourly → Kafka → Flink final window). Reduces Flink state size.
Cost: ~$35K/mo (linear growth from $22K)

At 10000 Metrics (24 months)

Real-time Sunset: Disable real-time path for non-critical metrics (e.g., internal diagnostics). Batch-only pipeline saves 40% Flink compute.
Federated Query: Multiple Snowflake clusters by region (US-east, EU, APAC). Reduces latency for regional queries.
Tiering: S3 → Glacier after 90 days (cold archive, rarely queried)

💰 COST BREAKDOWN

Current (1000 metrics, 100 users)

Total: ~$22K/month
• Kafka: $3K (6 brokers, HA)
• Flink: $8K (8-12 task mgrs)
• Redis: $5K (350GB cluster)
• Snowflake: $6K (2 XL, auto-suspend off-peak)

Cost Optimization Techniques

Tiered Storage: 7d Redis (hot, expensive), 30d Snowflake warm cluster, 2yr S3 + Glacier (cheap archive)
Compression: Parquet + Snappy = 10:1 ratio (100GB → 10GB stored)
Auto-Scaling: Flink scales on kafka_consumer_lag. Snowflake compute suspends off-peak (7 PM - 8 AM).
Pre-Aggregation: Materialized views reduce per-query compute 10x for golden metrics (indirect cost savings).

Cost Per Metric Per Month

$22K ÷ 1000 metrics = $22/metric/month
(At 5000 metrics: $35K ÷ 5000 = $7/metric/month. Scale improves unit economics!)

🎤 INTERVIEW NARRATIVE

"The core challenge here is that real-time and historical have fundamentally conflicting requirements.

Real-time needs <100ms latency—impossible with Snowflake alone. A query that scans 2 years of data (Snowflake's table) takes >500ms at best, even with vectorized execution. Historical needs 2+ years retention and analytical flexibility—Redis can't store that cheaply. Storing 2 years in RAM costs $200K+ annually.

So I use a dual storage approach: Redis for 7-day hot data with <5ms reads, Snowflake for cold data with <500ms queries. Materialized views pre-compute 50 golden metrics (revenue, orders, CSAT, etc.) at hourly and daily granularity, so 99% of queries hit pre-agg data and respond in <50ms.

For ingestion, I use Kafka with Schema Registry. Kafka gives replay capability (7-day retention) so if a stream job fails, I can re-process without re-running all aggregations. Schema Registry enforces producer-consumer contracts—prevents parsing failures when a metric's schema evolves (critical at 1000+ metrics).

For compute, I run two engines: Flink handles real-time streams with 15-30 second windows, flushed to Redis every minute (feeds dashboards instantly). Spark runs nightly batch jobs to re-aggregate the full day's events, catches late arrivals, and enforces exactly-once semantics for revenue metrics where accuracy is non-negotiable. Flink excels at latency, Spark excels at batch correctness. I don't force one to do both.

On the query side, I expose three APIs. Fast API hits Redis directly for real-time questions (<100ms). Analytical API hits Snowflake for complex joins (<500ms). GraphQL layer batches N independent requests into a single Snowflake query, reducing round-trips from N to 1.

The biggest trade-off is consistency. Because Redis and Snowflake are asynchronous, there's a ~5-minute lag where they diverge. For most metrics (pageviews, clicks), that's fine. For revenue, I run a nightly reconciliation that flags discrepancies >1% and alerts ops. It's a deliberate choice: buy latency by accepting eventual consistency rather than synchronous writes across both paths (which would add distributed transaction complexity).

At 100 concurrent users with 1000 metrics, this architecture costs ~$22K/month. If we grow to 5000 metrics, I'd shift 80% of hot data to Snowflake's hot cluster and keep only the top 200 metrics in Redis—trading some latency for linear cost growth to ~$35K/month."

❓ FOLLOW-UP QUESTIONS

Q: Why not just BigQuery/Redshift?
A: Can't achieve <100ms with cold storage alone. Columnar engines (BigQuery, Redshift) excel at full-table scans but struggle with sub-second latency. Tiered approach (hot + cold) is necessary.
Q: How do you handle schema evolution?
A: Schema Registry versions captured. Flink uses schema-on-read (ignores new fields, reads old ones). Spark handles multiple versions during backfill by selecting union of all schemas.
Q: What if Kafka goes down?
A: Apps buffer events locally for 1 hour. Kafka recovery takes <10min with proper replication (3x). Dashboards fall back to 7-day Redis cache (stale but available). Resume normal flow on recovery.
Q: How do you test dual-path consistency?
A: dbt tests on Snowflake (row counts, null checks), Flink unit tests, integration tests that write to both paths and compare tail records (last 100 rows must match).
Q: How do you handle hot metric skew?
A: Kafka partitioning by metric_id isolates hot metrics (revenue) from cold ones. "Revenue" partition stays on fast broker, doesn't block "page_views_on_mobile_in_japan" partition.
Q: Would you use Kinesis/BigQuery?
A: Kinesis ($10K/mo) costs 3x Kafka. BigQuery + Cloud Spanner have vendor lock-in. Kafka + Snowflake stack is open, cheaper, flexible for multi-cloud future.
