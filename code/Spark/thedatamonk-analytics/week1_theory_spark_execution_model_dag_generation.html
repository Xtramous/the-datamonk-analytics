🏗️1. Spark Architecture: The Big Picture

1.1 Core Components Overview

Spark operates on a master-worker architecture. Let's break it down:

Spark Cluster Architecture
    ┌─────────────────────────────────────────────────────────┐
    │                    YOUR APPLICATION                     │
    │                  (Python/Scala Script)                  │
    └────────────────────┬────────────────────────────────────┘
                         │
                         ▼
    ┌─────────────────────────────────────────────────────────┐
    │              DRIVER PROCESS                             │
    │                                                         │
    │  • Reads your code                                     │
    │  • Creates SparkSession                               │
    │  • Plans execution (DAG)                              │
    │  • Sends tasks to executors                           │
    │  • Collects results                                   │
    │                                                         │
    │  (In local mode: runs on YOUR machine)                │
    └────────────────────┬────────────────────────────────────┘
                         │
          ┌──────────────┼──────────────┐
          ▼              ▼              ▼
    ┌──────────────┬──────────────┬──────────────┐
    │  EXECUTOR 1  │  EXECUTOR 2  │  EXECUTOR 3  │
    │              │              │              │
    │ • Runs tasks │ • Runs tasks │ • Runs tasks │
    │ • Processes  │ • Processes  │ • Processes  │
    │   data       │   data       │   data       │
    │ • Returns    │ • Returns    │ • Returns    │
    │   results    │   results    │   results    │
    │              │              │              │
    │ (CPU threads)│ (CPU threads)│ (CPU threads)│
    └──────────────┴──────────────┴──────────────┘

    Note: In LOCAL MODE on your laptop, all these run on ONE machine!
In Local Mode (Your Setup):
When you run Spark locally:
  • Driver: One process (your Python script)
  • Executors: Multiple threads on the same machine (default: number of CPU cores)
  • Memory: Shared from your machine's RAM
  • Purpose: Simulates distributed execution on a single machine
Example: If your Mac has 8 CPU cores → 8 executor threads work in parallel.

1.2 Key Components Deep Dive

Component Role Analogy In Your Code
Driver Orchestrates execution; runs main code Project Manager Your Python script (running)
Executor Processes data in parallel; runs tasks Worker (multiple workers) CPU threads processing partitions
Partition Chunk of data; smallest unit of parallelism Task assignment to worker DataFrame split into N partitions
Task Work on one partition by one executor Specific job for one worker Process one partition of data
Stage Set of tasks that can run in parallel Phase of project Execution phase (e.g., "read data")
Job All stages needed for one action Complete project All work triggered by `.show()`
💡 Key Insight: Partitions = Parallelism
If your DataFrame has 8 partitions and 8 executors (threads):
  • Each partition is processed by one executor in parallel
  • All 8 partitions are processed simultaneously
  • Total time = max(time for any partition) [not sum of all times]
More partitions ≠ always faster! Too many partitions = overhead. Sweet spot: 2-3 partitions per executor core.

🔗2. RDD Lineage & Fault Tolerance

2.1 What is an RDD?

RDD = Resilient Distributed Dataset

RDD = Three Properties
    RESILIENT (Resilient)
    │
    ├─ Fault-tolerant
                ├─ If a partition is lost, Spark can recompute it
                └─ Achieved through LINEAGE (remember where it came from)
    │
    DISTRIBUTED (Distributed)
    │
    ├─ Data split across partitions
                └─ Can be processed in parallel on multiple nodes/threads
    │
    DATASET (Dataset)
    │
    └─ Immutable collection of objects
        └─ Cannot change; only create new RDDs through transformations
Why This Matters for L5:
Understanding RDDs helps you understand how Spark recovers from failures. At L5 interviews, you'll be asked: "If an executor crashes during job execution, how does Spark recover?" The answer: via RDD lineage.

2.2 RDD Lineage (The Critical Concept)

Lineage = Recipe for recreating a partition if lost

RDD Lineage Example
    Step 1: Read CSV file
    ┌─────────────────────┐
    │  RDD0 (CSV data)    │  Lineage: Read from "data.csv"
    │ Partitions: [0,1,2] │
    └──────────┬──────────┘
               │
    Step 2: Filter (transformation)
               ▼
    ┌─────────────────────┐
    │  RDD1 (filtered)    │  Lineage: Filter RDD0 where age > 25
    │ Partitions: [0,1,2] │
    └──────────┬──────────┘
               │
    Step 3: Map (transformation)
               ▼
    ┌─────────────────────┐
    │  RDD2 (mapped)      │  Lineage: Map RDD1 to (name, salary)
    │ Partitions: [0,1,2] │
    └──────────┬──────────┘
               │
    Step 4: Action (collect) - EXECUTION HAPPENS HERE
               ▼
    ┌─────────────────────┐
    │  Results returned   │
    └─────────────────────┘

    If executor crashes during step 3:
    • Spark knows: "RDD2 partition 1 was lost"
    • Spark recomputes: Re-read CSV → Filter → Map for partition 1
    • No data loss!
⚠️ Important Distinction
Wide vs Narrow Lineage:
  • Narrow: One partition of parent RDD → one partition of child (e.g., map, filter)
  • Wide: Many partitions → one partition (e.g., groupBy, join) = SHUFFLE
Why? Wide transformations require shuffle (data movement). We'll cover this next!

⚡3. Transformations vs Actions: Lazy Evaluation

3.1 The Core Concept: Lazy Evaluation

Transformations are lazy. Actions trigger execution.

Lazy Evaluation Timeline
    CODE WRITTEN                            EXECUTION HAPPENS
    ───────────────────────────────────     ─────────────────

    df.read.csv(...)      ← Transformation   (planning)
    df.filter(...)        ← Transformation   (planning)
    df.select(...)        ← Transformation   (planning)
    df.groupBy(...)       ← Transformation   (planning)
    df.agg(...)           ← Transformation   (planning)

    At this point: NOTHING HAS EXECUTED YET!
    Spark just built a PLAN (DAG).

                                             ▼
    df.show()             ← ACTION           EXECUTION!
    or                                       (driver triggers
    df.collect()          ← ACTION           all tasks)
    or
    df.write()            ← ACTION

    Result: Actual data processing happens here
Why Lazy Evaluation?
Benefit: Spark can optimize the entire plan before executing.
  • Example: If you filter first, then select only 2 columns, Spark can push the selection into the filter stage
  • Result: Faster execution, less data movement
  • This optimization is called Catalyst Optimizer (Week 2 topic)

3.2 Transformation vs Action: Complete List

Transformation (Lazy) Action (Eager)
map() show()
filter() collect()
select() write()
groupBy() count()
join() first()
union() take(n)
distinct() saveAsTable()
withColumn() rdd.collect()
📝 Example: Lazy Evaluation in Action
# Python code df = spark.read.csv("data.csv", header=True) # ↑ No data read yet! Just a plan. filtered_df = df.filter(df.age > 25) # ↑ Still no data filtering! Just updated the plan. selected_df = filtered_df.select("name", "salary") # ↑ Still no execution! Plan now includes: read → filter → select. result = selected_df.show() # ↑ EXECUTION HAPPENS HERE! # Spark now: # 1. Reads CSV # 2. Filters where age > 25 # 3. Selects only name, salary # 4. Displays results # All in ONE execution!

🔍4. Job → Stage → Task Breakdown

4.1 Hierarchy Explained

Spark Execution Hierarchy
    ONE ACTION (e.g., .show())
           │
           ├─ Creates ONE JOB
           │
           └─ One Job has ONE+ STAGES
               │
               ├─ Stage 0: Read CSV → Filter → Select
               │
               ├─ Stage 1: Shuffle (if needed)
               │
               └─ Stage 2: Final aggregation
                   │
                   └─ One Stage has MANY TASKS
                       │
                       ├─ Task 0: Process partition 0
                       ├─ Task 1: Process partition 1
                       ├─ Task 2: Process partition 2
                       └─ Task 3: Process partition 3
                           │
                           └─ All tasks run in parallel!
                               (up to # of executors)
What Creates New Stages?
Shuffles create new stages!
  • groupBy() → Shuffle (Stage break)
  • join() → Shuffle (Stage break)
  • repartition() → Shuffle (Stage break)
  • distinct() → Shuffle (Stage break)
Why? Because data must move between partitions. Spark can't do that within a stage.

4.2 Real Example: Multi-Stage Job

# Code df = spark.read.csv("users.csv", header=True) # Stage 0: Read + Filter (narrow transformations) filtered = df.filter(df.age > 25) # Stage 1: GroupBy (SHUFFLE - new stage!) grouped = filtered.groupBy("city").count() # Stage 2: Sort (wide transformation - potential new stage) sorted_result = grouped.orderBy("count", ascending=False) # ACTION - Triggers execution of all stages sorted_result.show()
Execution Flow for Above Code
    JOB: sort_result.show()
    │
    ├─ STAGE 0 (Narrow):
    │  ├─ Task 0: Read partition 0 → Filter → select partition 0
    │  ├─ Task 1: Read partition 1 → Filter → select partition 1
    │  ├─ Task 2: Read partition 2 → Filter → select partition 2
    │  └─ Task 3: Read partition 3 → Filter → select partition 3
    │  └─ (All run in parallel)
    │
    ├─ SHUFFLE (data moved between partitions)
    │
    ├─ STAGE 1 (Wide - GroupBy):
    │  ├─ Task 4: Aggregate received data from all tasks 0-3
    │  ├─ Task 5: Aggregate received data from all tasks 0-3
    │  └─ (Fewer tasks due to fewer distinct cities)
    │
    ├─ SHUFFLE (if needed for sort)
    │
    └─ STAGE 2 (Sort):
       └─ Task 6: Sort final result and display

🔀5. Shuffle Operations: When & Why

5.1 What is a Shuffle?

Shuffle = Moving data between partitions

Shuffle Mechanics
    BEFORE SHUFFLE:
    ┌──────────────────┐
    │  Partition 0     │  contains: rows with age 20, 21, 22
    └──────────────────┘
    ┌──────────────────┐
    │  Partition 1     │  contains: rows with age 23, 24, 25
    └──────────────────┘
    ┌──────────────────┐
    │  Partition 2     │  contains: rows with age 26, 27, 28
    └──────────────────┘

    When you do: df.groupBy("city")

    Spark needs to group ALL rows from same city together!
    But they're scattered across partitions 0, 1, 2!

    SHUFFLE PROCESS:
    1. Map Phase: Each partition hashes rows by city
    2. Shuffle: Data moves over network/disk
    3. Reduce Phase: New partitions collect grouped data

    AFTER SHUFFLE:
    ┌──────────────────┐
    │  Partition 0     │  contains: all rows from city "NYC"
    └──────────────────┘
    ┌──────────────────┐
    │  Partition 1     │  contains: all rows from city "LA"
    └──────────────────┘
    ┌──────────────────┐
    │  Partition 2     │  contains: all rows from city "SF"
    └──────────────────┘

    Cost: HIGH! (Network I/O, Disk I/O, Memory usage)
⚠️ Shuffles Are Expensive!
Why? Every byte of data must be written to disk, sent over network, and read back. This is 100x slower than in-memory operations!

At L5 level: You'll be asked to identify shuffles in queries and suggest optimizations to avoid them.

5.2 Which Operations Trigger Shuffles?

Operation Shuffle? Why? Example
filter() ❌ No Data stays in same partition df.filter(df.age > 25)
map() ❌ No 1-to-1 transformation per partition df.map(lambda x: x*2)
groupBy() ✅ Yes Must collect same keys together df.groupBy("city").count()
join() ✅ Yes Must align data from both tables df1.join(df2, on="id")
orderBy() ✅ Yes Must rearrange all data df.orderBy("salary")
distinct() ✅ Yes Must find duplicates globally df.distinct()
union() ❌ No Just concatenates partitions df1.union(df2)
💡 Optimization Insight
Minimize shuffles! Example:
  • ❌ Bad: df.groupBy("city").count().orderBy("count") → 2 shuffles
  • ✅ Better: df.groupBy("city").count().orderBy("count", ascending=False) → 2 shuffles (same, but more explicit)
  • ✅ Best: Depends on data size; sometimes 2 shuffles is unavoidable!

📈6. DAG (Directed Acyclic Graph) Visualization

6.1 What is a DAG?

DAG = Spark's execution plan visualized

DAG Properties:
  • Directed: Follows data flow direction (top to bottom)
  • Acyclic: No cycles (can't go back up)
  • Graph: Nodes = RDDs/DataFrames; Edges = transformations
Simple DAG Example
    Code:
    df = spark.read.csv("data.csv")
    filtered = df.filter(df.age > 25)
    selected = filtered.select("name", "salary")
    result = selected.collect()

    DAG Visualization:
    ┌──────────────────────┐
    │   CSV File           │
    │   (External)         │
    └──────────┬───────────┘
               │ read()
               ▼
    ┌──────────────────────┐
    │   RDD0               │
    │   (raw data)         │
    └──────────┬───────────┘
               │ filter(age > 25)
               ▼
    ┌──────────────────────┐
    │   RDD1               │
    │   (filtered)         │
    └──────────┬───────────┘
               │ select(name, salary)
               ▼
    ┌──────────────────────┐
    │   RDD2               │
    │   (selected columns) │
    └──────────┬───────────┘
               │ collect()
               ▼
    ┌──────────────────────┐
    │   Results in Memory  │
    │   (action executed)  │
    └──────────────────────┘

6.2 Reading Spark UI DAG Visualization

When you run a Spark job and open localhost:4040, you see the DAG. Here's how to read it:

Spark UI DAG (Typical Structure)
    │
    │  (Click "Jobs" → click on a Job → "DAG Visualization")
    │
    │  ┌───────────────────────────────────────┐
    │  │  Stage 0                              │
    │  │  ┌─────────────────────────────────┐  │
    │  │  │  CSVFileRDD (read)              │  │
    │  │  │  4 partitions                   │  │
    │  │  └────────────┬────────────────────┘  │
    │  │               │ map (read logic)      │
    │  │  ┌────────────▼────────────────────┐  │
    │  │  │  MapPartitionsRDD (filter)      │  │
    │  │  │  4 partitions                   │  │
    │  │  └────────────┬────────────────────┘  │
    │  │               │ map (select logic)    │
    │  │  ┌────────────▼────────────────────┐  │
    │  │  │  MapPartitionsRDD (select)      │  │
    │  │  │  4 partitions                   │  │
    │  │  └─────────────────────────────────┘  │
    │  └───────────────────────────────────────┘
    │                   │
    │                   │ (no shuffle, same stage)
    │                   │
    │  ┌───────────────────────────────────────┐
    │  │  Stage 1                              │
    │  │  ┌─────────────────────────────────┐  │
    │  │  │  collect()                      │  │
    │  │  │  1 task (reduce phase)          │  │
    │  │  └─────────────────────────────────┘  │
    │  └───────────────────────────────────────┘

    Reading Tips:
    • Boxes = RDDs
    • Arrows = transformations
    • Stage breaks = shuffles (invisible in diagram but indicated by task count change)
    • Click stage to see tasks, execution time, memory usage
🔍 What to Look For in DAG
  • Number of stages: Fewer stages = fewer shuffles = faster
  • Partition count: If drops suddenly, shuffle happened
  • RDD types: MapPartitionsRDD (narrow), ShuffledRDD (wide)
  • Stage timing: Which stage took longest? Opportunity to optimize

🎯 Key Points to Remember
  • Driver + Executors: Driver orchestrates; Executors do the work in parallel
  • Lazy Evaluation: Transformations are lazy (just planning); Actions trigger execution
  • Partitions = Parallelism: More partitions (up to executor count) = faster execution
  • RDD Lineage: Recipe for recomputing lost partitions; enables fault tolerance
  • Job → Stage → Task: Hierarchy of execution; shuffles create stage boundaries
  • Shuffles are expensive: Avoid them when possible; identify them in DAGs
  • DAG visualization: Use Spark UI to understand execution plan and identify bottlenecks
  • Narrow vs Wide: Narrow = 1-to-1 partition mapping; Wide = many-to-one (shuffle)
